# UML·동작 명세 가이드

## 1. 목적

Draw.io를 사용한 UML 다이어그램에서 **설비 동작/시퀀스** 정보를 추출하고, 코드 생성(Mechanical / Control / Process 레이어)에 일관되게 반영하기 위한 규칙을 정의한다.

---

## 2. 사용할 UML 요소

동작 정의의 주 소스로 사용할 다이어그램 종류를 정한다.

| 다이어그램 | 용도 | 생성 대상 레이어 |
|------------|------|-------------------|
| **상태 다이어그램** | 설비 전체 상태·전이 | Process Layer (상태머신, 시퀀스) |
| **시퀀스 다이어그램** | 단위/복합 동작 순서, 메시지 흐름 | Control Layer, Process Layer |
| **액티비티 다이어그램** | 복합 시퀀스, 분기·병합 | Control Layer, Mechanical Layer |

- **상태/시퀀스** → Process Layer (스레드·상태머신·전체 시퀀스).
- **복합 흐름** → Control Layer (Mechanical 조합).
- **단위 동작/유닛** → Mechanical Layer (유닛 클래스·기본 메서드).

---

## 3. Draw.io → 구조화 데이터

### 3.1 추출 절차

- Draw.io 파일(.drawio)은 XML 기반이므로, **파싱 규칙**을 정하여 텍스트/구조를 추출한다.
- 추출 대상: 도형/라인의 **텍스트**, **스테레오타입**, **태그** (팀에서 정의한 키워드).

### 3.2 서비스에서의 처리

- **미정**: 서버에서 Draw.io XML을 파싱한 뒤 텍스트/구조만 LLM에 넘길지, 사용자가 미리 “동작 설명 텍스트”를 입력할지 설계 단계에서 확정.

---

## 4. 네이밍·키워드 규칙

- **액터/메시지/상태 이름**과 실제 제어 단계(축 이동, IO ON/OFF, 로봇 명령 등)의 **매핑 규칙**을 문서화한다.
- 예: 상태명 `Loading` → enum `EquipmentState.Loading`, 메서드명 `DoLoading()` 등.
- 팀 내에서 동일한 키워드를 사용하면 LLM 프롬프트와 생성 코드의 일관성이 높아진다.

---

## 5. 예시 및 레이어 매핑

- **예시 다이어그램**: 1~2개 설비 시나리오에 대한 샘플 Draw.io와, “이렇게 그리면 이 코드가 나온다”는 대응 예시를 이 문서 또는 별도 예제 폴더에 두는 것을 권장한다.
- **레이어 매핑**:
  - 상태/시퀀스 다이어그램 → Process Layer.
  - 복합 시퀀스/액티비티 → Control Layer.
  - 단위 동작/유닛 정의 → Mechanical Layer.
