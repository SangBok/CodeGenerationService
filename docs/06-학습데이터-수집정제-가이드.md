# 학습 데이터 수집·정제 가이드

## 1. 목적

설비 제어 코드 자동 생성 LLM 서비스의 Few-shot 또는 파인튜닝을 위해, **기존 설계 자료와 코드의 쌍**을 수집·정제하는 절차와 포맷을 정의한다.

---

## 2. 쌍(pair) 정의

- **문서**: 하드웨어 요약 + 동작 설명(또는 UML 요약 텍스트). 설비별로 어떤 하드웨어가 있고, 어떤 동작/시퀀스를 하는지가 담긴 텍스트 또는 구조화 데이터.
- **코드**: 해당 설비용 C# 제어 코드 (레이어드 아키텍처에 맞는 소스).
- **레이어별 구분**: 가능하면 **레이어별**로 구분해 수집하면 Few-shot/파인튜닝 시 활용도가 높다.
  - 예: Mechanical 유닛 클래스만, Process 상태머신만, Control 복합 동작만 등.

---

## 3. 수집 절차

- 기존 설비별 폴더/저장소에서 **어떤 파일을 문서로, 어떤 파일을 코드로** 분류할지 기준을 정한다.
- **메타데이터** 기록 방법을 정한다.
  - 설비명, 제조사(또는 설비 타입), 수집 일자, 레이어 태그(System/Hardware/Mechanical/Control/Process) 등.
- 수집 시 **저작권·보안** (내부 전용, 외부 유출 금지 등) 정책을 따른다.

---

## 4. 정제 기준

- **개인정보·민감 정보**: 내부 IP, 담당자명 등 제거 또는 익명화.
- **코딩 스타일·네이밍**: 팀에서 정한 코딩 스타일/네이밍 규칙에 맞게 통일할지, 원본 유지할지 결정.
- **최소 품질**: 컴파일 가능 여부, 단위 테스트 유무 등을 기준으로 포함/제외할지 정한다.
- **타깃 프레임워크**: .NET 전환 예정이므로, 수집 코드가 .NET Framework인 경우 .NET 호환 여부나 마이그레이션 예시로 쓸지 정책을 정한다.

---

## 5. 저장 포맷

- **권장**: JSONL 등으로 (문서 블록, 코드 블록, 메타데이터) 저장 형식을 정의한다.
- 예시 구조:
  - `doc_snippet`: 설계 자료 요약 또는 UML 추출 텍스트.
  - `code_snippet`: 해당하는 C# 코드 조각 또는 전체 파일.
  - `metadata`: 설비명, 레이어, 파일 경로, 날짜 등.
- 파인튜닝 시 위 포맷이 학습 데이터로 쓰일 수 있도록 필드 구조를 맞춘다.

---

## 6. Qwen 파인튜닝 vs 프롬프트(Few-shot)

- **파인튜닝**: 위 포맷으로 구성한 데이터로 Qwen 모델을 파인튜닝할 경우, 필드 구조(입력/출력 형식)를 학습 목적에 맞게 고정한다.
- **Few-shot**: 파인튜닝 없이 프롬프트에 "설계 자료 조각 + 생성 코드 예시" 1~2건을 포함해 사용할 경우, 레이어별 예시를 준비한다.
- 프로젝트 단계에서 **파인튜닝을 할지, Few-shot만 쓸지** 정책을 정하고, 그에 맞춰 수집·포맷을 조정한다.
