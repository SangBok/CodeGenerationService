# 요구사항 명세서 (SRS)

## 1. 개요

### 1.1 목적

설비의 하드웨어 정보와 Draw.io UML 기반 동작 명세를 입력으로 받아, **설비 제어용 C# 코드**를 자동 생성하는 **LLM 서비스**를 개발하기 위한 요구사항을 정의한다.

### 1.2 범위

- **대상 서비스**: 설비 설계 자료(하드웨어 명세 + Draw.io UML)를 입력받아 레이어드 아키텍처에 맞는 C# 제어 코드를 생성하는 LLM 기반 코드 생성 서비스
- **생성 코드 타깃**: C# .NET (현대 .NET, 예: .NET 6/8). 현재 설비 제어 코드는 .NET Framework + WinForms 기반이나 **.NET으로 전환 예정**이며, 생성 코드는 .NET 타깃으로 통일
- **배포**: 온프레미스(로컬 서버) 환경
- **LLM**: Qwen 모델 사용 예정

---

## 2. 기능 요구사항

### 2.1 입력

| 입력 | 설명 |
|------|------|
| **하드웨어 명세** | 축 정보(개수, 타입, 제조사/프로토콜), IO 보드(종류, 개수, 채널), 로봇(제조사, 프로토콜, 개수), 네트워크/주소, 안전 IO 등 설비 하드웨어 정보 |
| **Draw.io UML** | 설비 동작 정의 — 상태 다이어그램, 시퀀스 다이어그램, 액티비티 다이어그램 등 (동작/시퀀스 명세) |

### 2.2 출력

- **레이어드 C# 제어 코드**: System → Hardware → Mechanical → Control → Process 5계층 구조에 맞는 C# 소스 코드
- 생성 코드는 .NET 타깃, WinForms on .NET 등 전환 후 스택에 맞는 라이브러리만 참조

### 2.3 사용자 시나리오

1. 사용자가 설비의 **하드웨어 명세**(구조화된 형식)를 업로드 또는 입력한다.
2. 사용자가 설비 **동작/시퀀스**를 정의한 **Draw.io UML** 파일(또는 추출된 텍스트)을 업로드 또는 입력한다.
3. 서비스는 위 입력을 바탕으로 **레이어별 단계적** 코드 생성을 수행한다 (System → Hardware → Mechanical → Control → Process).
4. 사용자는 생성된 C# 프로젝트/파일을 받아 검토·수정·배포한다.

### 2.4 학습 데이터

- 기존에 작성된 **설계 자료와 코드의 쌍**을 수집하여 Few-shot 또는 파인튜닝에 활용한다.
- 레이어별로 구분된 학습 데이터 수집이 권장된다.

---

## 3. 비기능 요구사항

### 3.1 기술 환경

- **생성 코드 타깃**: C# .NET (예: .NET 6 또는 .NET 8)
- **현재 설비 제어 코드**: .NET Framework + WinForms 동작 중 → **.NET 전환 예정**
- **배포**: 온프레미스

### 3.2 미정/추가 확정 필요 사항

- 인증/권한: 온프레미스 전용 여부, 내부 인증(AD/LDAP 등) 필요 여부
- Draw.io 처리: 서버에서 Draw.io XML 파싱 후 LLM에 넘길지, 사용자가 동작 설명 텍스트를 미리 입력할지
- 생성 코드 배포 형태: 솔루션/프로젝트 전체 생성, 기존 프로젝트에 파일만 추가, 스니펫만 등
- 생성 코드 버전/이력 보관 및 저장소 정책
- 생성 코드 품질 검증: 정적 분석(린트), 컴파일 체크, 단위 테스트 자동 생성 여부
